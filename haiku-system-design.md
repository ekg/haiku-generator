# Haiku Observation Agent System — Design Document

## Overview

An autopoietic system that wakes every 5 minutes, fans out three observer agents
in parallel, and each writes a haiku about the current state of the machine.
The system is self-sustaining: a workgraph cycle drives the loop, and observer
tasks are created fresh each iteration.

## Loop Mechanism: Workgraph Cycle

**Recommended approach: workgraph structural cycle with `--cycle-delay 5m`.**

### Why workgraph cycles (not cron, not shell loops)

| Approach | Pros | Cons |
|----------|------|------|
| **wg cycle** (recommended) | Native to workgraph; agents dispatched automatically; logs/artifacts tracked; convergence detection; no external dependencies | Requires cycle-aware task structure |
| Cron job | Simple, OS-level | No workgraph visibility; haiku tasks invisible to `wg status`; separate monitoring |
| Shell sleep loop | Simplest | Fragile; dies on terminal close; no recovery; no workgraph integration |

### Cycle Structure

The cycle is a self-referencing task chain:

```
observe-haiku-cycle (cycle header, --max-iterations 0, --cycle-delay 5m, --no-converge)
  ├── disk-observer-N    (parallel)
  ├── process-observer-N (parallel)
  ├── network-observer-N (parallel)
  └── aggregate-haikus-N (--after all three observers)
        └── back-edge to observe-haiku-cycle
```

- `--max-iterations 0` means unlimited iterations (runs forever until manually stopped)
- `--cycle-delay 5m` enforces a 5-minute pause between iterations
- `--no-converge` prevents agents from stopping the cycle early
- Each iteration creates fresh observer tasks; the aggregator depends on all three

### Bootstrap Command

```bash
# Create the cycle header
wg add "Haiku observation cycle" \
  --id haiku-cycle \
  --max-iterations 0 \
  --cycle-delay 5m \
  --no-converge \
  -d "Recurring 5-minute cycle that fans out observer agents to write haikus about system state." \
  -t haiku,cycle

# The cycle body is created by the cycle header task's agent each iteration.
# See "Workgraph Task Templates" below for the per-iteration tasks.
```

### Naming Convention

Haiku files: `{observer}-{YYYYMMDD-HHMMSS}.haiku`

- Observer types: `disk`, `process`, `network`
- Timestamp is UTC at observation time
- Extension `.haiku` makes them easy to glob

## Observer Agent Designs

Each observer is a workgraph task with `--exec-mode shell` (runs a script, no LLM needed
for the observation itself — the haiku is generated by the LLM agent executing the task).

Alternatively, each observer is a lightweight LLM task that:
1. Runs system commands to gather data
2. Composes a haiku from the observations
3. Writes the haiku file atomically

**Recommended: LLM-based observers** — haiku composition benefits from language model creativity.
Each observer task uses `--exec-mode full` with a small/fast model.

### Disk Observer

**What it observes:**
- `df -h` — disk usage percentages and mount points
- `du -sh /home/erik/autohaiku/haikus/` — haiku directory size growth
- `find /home/erik -maxdepth 2 -mmin -5 -type f 2>/dev/null | head -10` — recently changed files
- `ls -lt /home/erik/autohaiku/haikus/ | head -5` — most recent haikus

**Haiku theme:** Storage, fullness, emptiness, accumulation, filesystem as landscape.

### Process Observer

**What it observes:**
- `uptime` — load averages
- `ps aux --sort=-%mem | head -6` — top memory consumers
- `ps aux --sort=-%cpu | head -6` — top CPU consumers
- `free -h` — memory usage
- `nproc` — CPU count

**Haiku theme:** Labor, rest, burden, breathing, processes as living beings.

### Network Observer

**What it observes:**
- `ip addr show | grep 'inet '` — IP addresses
- `ping -c 1 -W 2 8.8.8.8 2>&1` — connectivity and latency
- `ss -tunp | head -10` — active connections
- `ip route show default` — default gateway
- `cat /proc/net/dev | tail -n +3` — interface traffic counters

**Haiku theme:** Connection, distance, flow, packets as travelers, latency as longing.

### Observer Task Model Choice

Observers use `openrouter:deepseek/deepseek-v3.2` (fast tier, explicit provider) because:
- Haiku composition is a lightweight creative task
- Fast model minimizes cycle latency
- Cost is negligible per observation

## Atomicity Strategy

**Approach: temp file + atomic rename.**

Each observer writes its haiku using this pattern:

```bash
# 1. Write to temp file in the same directory (same filesystem = atomic rename)
TMPFILE="/home/erik/autohaiku/haikus/.disk-$TIMESTAMP.haiku.tmp"
FINAL="/home/erik/autohaiku/haikus/disk-$TIMESTAMP.haiku"

cat > "$TMPFILE" << 'HAIKU'
---
observer: disk
timestamp: 20260323-235500
---
Ninety percent full
Silent blocks await their fate
Delete or expand
HAIKU

# 2. Atomic rename
mv "$TMPFILE" "$FINAL"
```

### Why this approach

- **No lock contention**: Each observer writes a unique filename — no shared file.
- **Atomic rename**: `mv` on the same filesystem is atomic (single `rename(2)` syscall).
- **Crash safety**: If the agent dies mid-write, only a `.tmp` file is left, never a partial `.haiku`.
- **No flock needed**: Since files don't collide, there's nothing to lock.
- **Cleanup**: A periodic `find haikus/ -name '*.tmp' -mmin +10 -delete` can clean orphaned temps.

## Integration with Downstream Tasks

The three implementation tasks (`implement-disk-observer`, `implement-process-observer`,
`implement-network-observer`) should each:

1. Create the observer script in `scripts/{observer}-observer.sh`
2. Implement the observation commands listed above
3. Handle the atomic write pattern (tmp + mv)
4. Include the YAML frontmatter in output
5. Register the script path as a workgraph artifact

The cycle header task and aggregator can be implemented as a follow-up once all
three observers are working.

## Stopping the System

```bash
# Graceful stop: let current iteration finish, then stop
wg done haiku-cycle --converged

# Immediate stop (if --no-converge was set, use cancel instead)
wg fail haiku-cycle --reason "Manual stop requested"
```
